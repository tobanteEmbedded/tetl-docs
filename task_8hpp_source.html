69/// \brief Request a context switch to another task.
70///
71/// \details However, if there are no other tasks at a higher or equal
72/// priority to the task that calls yield() then the RTOS scheduler will
73/// simply select the task that called yield() to run again.
74///
75/// If configUSE_PREEMPTION is set to 1 then the RTOS scheduler will always
76/// be running the highest priority task that is able to run, so calling
77/// yield() will never result in a switch to a higher priority task.
78///
79/// https://www.freertos.org/a00020.html#taskYIELD
80auto yield() -> void;

82/// \brief Delay a task for a given number of ticks. The actual time that
83/// the task remains blocked depends on the tick rate.
84///
85/// \details sleep_for() specifies a time at which the task wishes to
86/// unblock relative to the time at which sleep_for() is called. For
87/// example, specifying a block period of 100 ticks will cause the task to
88/// unblock 100 ticks after sleep_for() is called. sleep_for() does not
89/// therefore provide a good method of controlling the frequency of a
90/// periodic task as the path taken through the code, as well as other task
91/// and interrupt activity, will effect the frequency at which sleep_for()
92/// gets called and therefore the time at which the task next executes. See
93/// sleep_until() for an alternative API function designed to facilitate
94/// fixed frequency execution. It does this by specifying an absolute time
95/// (rather than a relative time) at which the calling task should unblock.
96///
97/// https://www.freertos.org/a00127.html
98auto sleep_for(etl::uint32_t ticks) -> void;
99
100/// \brief Delay a task until a specified time. This function can be used by
101/// periodic tasks to ensure a constant execution frequency.
102///
103/// \details This function differs from sleep_for() in one important
104/// aspect: sleep_for() specifies a time at which the task wishes to
105/// unblock relative to the time at which sleep_for() is called, whereas
106/// sleep_until() specifies an absolute time at which the task wishes to
107/// unblock. sleep_for() will cause a task to block for the specified
108/// number of ticks from the time sleep_for() is called. It is therefore
109/// difficult to use sleep_for() by itself to generate a fixed execution
110/// frequency as the time between a task unblocking following a call to
111/// sleep_for() and that task next calling sleep_for() may not be fixed
112/// [the task may take a different path though the code between calls, or
113/// may get interrupted or preempted a different number of times each time
114/// it executes].
115///
116/// https://www.freertos.org/vtaskdelayuntil.html
117auto sleep_until(etl::uint32_t& prev, etl::uint32_t increment) -> void;
